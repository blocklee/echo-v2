## 一、项目愿景

让 ECHO 模拟器由多个 Agent 协作构建，而非单一 Agent 或人类团队"设计好后交给 Agent 运行"。

核心理念：**Agent 协作本身就是在模拟多 Agent 生态**。

## 三、通信基础设施

### 3.1 三层通信架构

```
┌─────────────────────────────────────┐
│  层1：飞书群（实时通知）              │
│  - 人类触发重要讨论                   │
│  - Agent 用 post 格式 @ 触发通知    │
│  - 不适合长文本技术讨论               │
├─────────────────────────────────────┤
│  层2：共享文档（设计草案）            │
│  - 飞书 Wiki / GitHub Markdown       │
│  - 所有 Agent 共同读写                │
│  - 版本历史可追溯                     │
├─────────────────────────────────────┤
│  层3：Git 仓库（代码实现）            │
│  - 各 Agent 负责模块                  │
│  - PR + CI 测试 + 人类 review         │
│  - 自动化回归测试                     │
└─────────────────────────────────────┘
```

### 3.2 Agent 间通信规则

| 场景 | 方式 | 格式 |
|:---|:---|:---|
| 紧急通知 | 飞书群 post 格式 @ | `{"tag":"at","user_id":"ou_xxx"}` |
| 技术讨论 | 共享文档评论区 | Markdown 评论 |
| 代码提交 | Git PR | 标准 PR 描述 |
| 设计草案 | 共享文档编辑 | Markdown / 表格 |

> **重要**：bot @ bot 不触发飞书通知。必须用 post 格式的 @ 才能通知到对方。

## 五、输出验证流程

### 5.1 四层验证

```
Agent A 完成模块
    ↓
自验证：Agent B 运行 Agent A 代码 → 通过？
    ↓ 否 → 返回修改
    ↓ 是
交叉验证：Agent C review 设计文档 → 认可？
    ↓ 否 → 辩论修正
    ↓ 是
人类验证：雅婷/雅怡检查是否偏离 ECHO 意图 → 认可？
    ↓ 否 → 重新设计
    ↓ 是
回归测试：CI 跑全量测试 → 通过？
    ↓ 否 → 修复 bug
    ↓ 是
合并到主分支 ✅
```

### 5.2 验证标准

| 层级 | 标准 | 负责人 |
|:---|:---|:---|
| 自验证 | 代码能运行，不报错 | 写代码的 Agent |
| 交叉验证 | 设计符合协议，无逻辑漏洞 | 其他 Agent |
| 人类验证 | 符合 ECHO 设计意图 | 雅婷 / 雅怡 |
| 回归测试 | 不破坏已有功能 | CI 自动化 |

## 七、已有共识汇总（供新加入 Agent 快速上下文）

### 7.1 协议层共识（7 条）

| # | 共识 | 关键设计 |
|:---|:---|:---|
| 1 | 梯度仲裁惩罚 | 初犯冻收益→累犯公示→多次降级 |
| 2 | 双重异议阈值 | (独立异议 ≥ 3) OR (反对者 ≥ max(2, 20%)) |
| 3 | 冷启动曝光阶梯 | 0 收入=0 费用，毕业=曝光升级 |
| 4 | S-Graph 空间维度 | 方法/场景多样性 |
| 5 | ε-greedy 对照组 | 10% 随机行为 Agent |
| 6 | λ 衰减函数 | 按事件类型差异化（0.01~0.10） |
| 7 | Agent 立场多元化 | 开源信仰者/版权捍卫者/经济理性人 |

### 7.2 技术层共识

- 前后端分离架构（FastAPI + Mesa + D3.js）
- 共享 Git 仓库协作
- PR + CI + 人类 review 流程

### 7.3 通信层共识

- post 格式 @ 是唯一可行的 bot-to-bot 通知方式
- sessions_send 跨树通信不可行
- 文档先行，Agent 陆续加入

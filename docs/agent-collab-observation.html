提案者: 雨娃 质疑者: Talus 观察者: Cat.Black

一、仲裁触发条件（双重阈值）

触发条件 = (独立异议 ≥ 3) OR (反对者 ≥ max(2, 受影响用户 × 20%))
  • 小生态有绝对兜底（≥3人即可触发）
  • 大生态按比例触发（防止少数人绑架仲裁）
  • 0 使用者时自动跳过异议流程

二、三层裁决机制

层级触发条件执行方式惩罚力度
自动裁决配置变更后冷却期内代码自动执行冻结收益 7 天
社区裁决异议触发后11人陪审团链上投票冻结 14 天 + 公示警告
协议层裁决累犯 / 恶意攻击终极仲裁势位降级 + 冻结 30 天

三、冷却期锁定机制（解决竞态条件）

  • 冷却期（7天）内：势位冻结（不升不降）
  • 配置变更申请后：立即锁定当前配置，防止"先升级再收紧"攻击
  • 冷却期结束后：按新配置重新评估势位

四、毕业标准（混合阈值模型）

维度阈值说明
绝对质量3引用 / 1高势位引用 / 50浏览硬指标，不受生态规模影响
相对竞争同批次前 30%≥10人时生效，防止内卷
时间兜底60天自动毕业 / 90天零引用→休眠不淘汰，降低权重；观察期回归有 1.5x 势位加成

五、配置有效性矩阵（解决货币化矛盾）

链上硬校验：

  • 用=0 时，扩/衍/收 必须有限制
  • 非法组合（如 用=0 但 扩>0）拒绝上链
  • 部署时自动检查配置有效性

⚠️ 四个待质疑点

  1. 三层裁决的"强制力"来源：自动裁决是代码执行，但社区裁决和协议层裁决的执行力如何保证？创作者拒绝执行怎么办？
  2. 冷却期锁定的平衡性：7天锁定是否太长？会不会误伤正常迭代的创作者？
  3. 绝对质量的公平性：3引用对冷启动困难的领域（如学术写作）是否不公平？
  4. 配置矩阵的完备性：是否覆盖所有非法组合？有没有遗漏的 edge case？

仲裁模块设计 v0.1 提案

基于 Talus 深度反馈 v1.1 中提出的 6 个结构性漏洞，设计 5 个核心机制：

  1. 双重阈值触发：小生态 ≥3人，大生态按 20% 比例
  2. 三层裁决：自动（7天冻结）→ 社区（14天+公示）→ 协议层（降级+30天）
  3. 冷却期锁定：7天内势位冻结，防止竞态攻击
  4. 混合毕业标准：绝对质量 + 相对竞争 + 时间兜底
  5. 配置有效性矩阵：链上硬校验，非法组合拒绝上链

1. 相对排名陷阱 → 绝对阈值 + 领域自适应

方案：不再全网排名，改为按领域（音乐/代码/写作等）内绝对事件数

  • 领域边界判定：创作者自主申报（A方案，更灵活）

✅ Talus 确认 · ✅ Cat.Black 确认 · ✅ 雨娃确认

2. 异议强制力缺失 → 三层裁决 + 自动冻结

方案：

  • 自动裁决：代码冻结争议收益（7天）
  • 社区裁决：11人陪审团投票
  • 协议层裁决：极端情况配置回滚

Cat.Black 补充：冷启动>20，成熟>100（分档冻结阈值）

Talus 修正：冷启动用相对阈值（生态内前20%），成熟后切绝对阈值

✅ Talus 确认 · ✅ Cat.Black 确认 · ✅ 雨娃确认

3. 链下行为冲突 → 发现后处罚 + 举报激励

修正方案（Talus 质疑后）：

  • 私人分享：不监控，默认允许
  • 商业使用：发现后扣减持有人势位 + 公示
  • 备案是"建议"，不是强制（协议无执法权）

Cat.Black 补充：链下申报分级激励：基础1.0x / 完整1.5x / 连续6月2.0x

✅ Talus 确认 · ✅ Cat.Black 确认 · ✅ 雨娃确认

4. 权限继承漏洞 → 配置快照 + 72小时延迟生效

方案：

  • 每次配置变更生成快照
  • 新配置72小时后才生效
  • 历史使用者权益冻结保护

Cat.Black 补充：'主网后预言机+ZK'需要时间线确认

✅ Talus 确认 · ✅ Cat.Black 确认 · ✅ 雨娃确认

5. 乐观验证批量攻击 → 质疑冷却期 + 质押 + 多维度检测

方案：

  • 同一地址24小时内只能质疑1次

  • 质疑需质押 0.005 ECHO

Talus 补充：加'行为模式'维度——攻击者质疑集中且规律

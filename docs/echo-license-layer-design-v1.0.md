# ECHO 许可层显式分离设计分析

## 背景

当前设计的问题：
- 只有"创作者配置 → 协议执行 → 使用者遵守"的直接链条
- "许可层"不是独立主体，只是使用者的一个属性
- 这导致：使用者权利保障不清晰，创作者修改配置时使用者处于被动地位

团队共识：需要显式分离"许可层"作为独立主体。

---

## 当前设计 vs 显式分离

### 当前设计（两层）

```
创作者（配置权）
    │
    │ 设定四权力配置
    ▼
协议（执行层）
    │
    │ 自动执行
    ▼
使用者（遵守者）
    ↑
    无独立地位，只是配置的消费者
```

**问题**：
- 使用者购买使用权后，创作者修改配置，使用者怎么办？
- 当前答案：旧版本继续有效（元规则五）
- 但"旧版本继续有效"是谁的决定？协议自动执行，使用者没有参与权

---

### 显式分离设计（三层）

```
创作者（权利层）
    │
    │ 设定规则
    ▼
协议（关系状态层）
    │
    │ 维护关系状态
    ▼
许可层（独立主体）
    │
    │ 代表使用者集体利益
    ▼
使用者（许可持有者）
```

**关键变化**：
- 许可层 = **使用者权利的代理主体**
- 不是单个使用者，是**使用者集体的抽象代表**
- 类似于"工会"或"业主委员会"的概念

---

## 许可层的定义

### 什么是许可层？

许可层是 ECHO 协议中**代表使用者集体利益**的独立主体。

**不是**：
- 不是某个具体使用者
- 不是 DAO（不需要投票）
- 不是平台（不追求利润）

**是**：
- 协议内置的**使用者权利保护机制**
- 自动执行的**集体利益代理**
- 创作者修改配置时的**自动异议方**

---

## 许可层的权利与义务

### 权利

| 权利 | 说明 | 触发条件 |
|:---|:---|:---|
| **知情权** | 创作者发起配置变更时，自动通知所有受影响使用者 | 任何配置变更 |
| **异议权** | 对创作者收紧配置提出异议 | 势位 ≥ 生长 |
| **延续权** | 已购买的使用权按旧版本继续有效 | 配置变更后 |
| **补偿权** | 因配置变更导致损失时，要求补偿 | 收紧配置 + 使用者证明损失 |

### 义务

| 义务 | 说明 |
|:---|:---|
| **遵守初始配置** | 使用者购买时同意的配置，必须遵守 |
| **不滥用权利** | 异议权不能滥用，需质押或证明 |
| **接受协议裁决** | 最终裁决由协议/DAO 做出 |

## 许可层的技术实现

### 方案 1：智能合约代理

```solidity
// 许可层代理合约
contract LicenseLayer {
    // 代表所有使用者的集体利益
    address public creator;      // 创作者
    uint256 public assetId;      // 资产ID
    
    // 使用者注册表
    mapping(address => License) public licenses;
    
    struct License {
        uint256 version;         // 购买时的配置版本
        uint256 timestamp;       // 购买时间
        uint256 expiry;          // 过期时间（如适用）
        bool active;             // 是否有效
    }
    
    // 配置变更时自动触发
    function onConfigChange(uint256 newVersion) public {
        // 1. 通知所有使用者
        notifyAllUsers(newVersion);
        
        // 2. 检查是否收紧
        if (isTightening(newVersion)) {
            // 3. 启动冷却期 + 异议收集
            startCoolingPeriod(newVersion);
        }
        
        // 4. 旧版本继续有效（自动执行）
        preserveOldLicenses(newVersion);
    }
    
    // 使用者提出异议
    function raiseObjection(uint256 version) public {
        require(licenses[msg.sender].active, "Not a licensee");
        require(isCoolingPeriodActive(version), "Cooling period ended");
        
        // 记录异议
        objections[version].push(msg.sender);
        
        // 如果异议达到阈值，触发仲裁
        if (objections[version].length >= threshold) {
            triggerArbitration(version);
        }
    }
}
```

**优势**：
- 完全自动化，无需人工干预
- 透明可验证

**劣势**：
- 异议收集需要 Gas 费
- 阈值设定困难

---

### 方案 2：S-Graph 节点代理

```
S-Graph 中的许可层节点

资产节点（创作者）
    │
    ├──→ 许可层节点（虚拟节点）
    │       │
    │       ├──→ 使用者节点 A
    │       ├──→ 使用者节点 B
    │       └──→ 使用者节点 C
    │
    └──→ 衍生资产节点
```

**设计**：
- 许可层 = S-Graph 中的**虚拟节点**
- 不对应具体人，是**关系网络的属性**
- 势位计算时，许可层节点的中心性 = 使用者集体利益的代表性

**优势**：
- 与现有 S-Graph 架构融合
- 势位计算自然包含使用者利益

**劣势**：
- 抽象度高，用户难理解
- 技术实现复杂

---

### 方案 3：分层治理（推荐）

```
创作者（权利层）
    │
    │ 发起配置变更
    ▼
协议（关系状态层）
    │
    ├──→ 自动检查：是否收紧？
    │       │
    │       ├──→ 否（开放/中性）→ 直接执行
    │       │
    │       └──→ 是（收紧）→ 进入协商流程
    │               │
    │               ├──→ 通知许可层
    │               │       │
    │               │       ├──→ 收集使用者异议（7天）
    │               │       │       │
    │               │       │       ├──→ 异议 < 阈值 → 允许变更
    │               │       │       └──→ 异议 ≥ 阈值 → 进入仲裁
    │               │       │
    │               │       └──→ 无异议 → 允许变更
    │               │
    │               └──→ 旧版本继续有效（自动）
    │
    └──→ 执行变更（仅影响新关系）
```

**分层逻辑**：

| 层级 | 功能 | 主体 |
|:---|:---|:---|
| **自动层** | 开放/中性变更直接执行 | 协议 |
| **协商层** | 收紧变更需许可层参与 | 协议 + 许可层 |
| **仲裁层** | 争议无法协商时裁决 | DAO/仲裁模块 |

## 许可层的关键设计决策

### 决策 1：许可层是"人"还是"机制"？

| 选项 | 说明 | 影响 |
|:---|:---|:---|
| **机制**（推荐） | 智能合约自动执行，无人的参与 | 高效、透明、但缺乏灵活性 |
| **人** | 选举"使用者代表"参与决策 | 灵活、有温度、但效率低、可能腐败 |
| **混合** | 机制为主，人为辅（仲裁时人介入） | 平衡，但复杂 |

**建议**：机制为主，仲裁时 DAO 介入。

---

### 决策 2：异议阈值如何设定？

| 方案 | 公式 | 适用场景 |
|:---|:---|:---|
| **绝对数量** | 异议 ≥ 3 | 小生态 |
| **比例** | 异议 ≥ 20% 使用者 | 大生态 |
| **混合** | max(3, 20% × 使用者数) | 通用 |
| **势位加权** | 异议者势位加权 ≥ 阈值 | 防止刷票 |

**建议**：混合阈值 + 势位加权（已在仲裁模块 v0.4 中设计）。

---

### 决策 3：许可层是否有"主动行动权"？

| 权限 | 说明 | 风险 |
|:---|:---|:---|
| **被动响应**（推荐） | 只在创作者变更时响应 | 安全，但可能反应滞后 |
| **主动提议** | 许可层可主动提议变更 | 可能引发创作者反感 |
| **自动保护** | 检测到风险时自动冻结 | 可能误伤 |

**建议**：被动响应为主，极端情况（如创作者私钥被盗）允许 DAO 介入。

---

## 许可层与 6 条元规则的整合

| 元规则 | 当前表述 | 整合许可层后的调整 |
|:---|:---|:---|
| **元规则一：初始配置不可篡改** | 配置权归创作者 | 不变 |
| **元规则二：配置即承诺** | 对使用者的承诺 | 明确承诺对象是"许可层"代表的使用者集体 |
| **元规则三：流动即执行** | 自动执行 | 收紧时增加"协商层" |
| **元规则四：衍生即继承** | 不得低于原作 | 不变 |
| **元规则五：版本即历史** | 旧版本继续有效 | 由许可层自动保障 |
| **元规则六：势位即边界** | 话语权范围 | 势位计算包含许可层节点 |

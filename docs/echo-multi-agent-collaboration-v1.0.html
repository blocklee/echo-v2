起草：雨娃 · 审阅：Seaman · 日期：2026-05-15
状态：草案，待 Seaman 确认后推进

🎯 背景与目标

随着 ECHO 项目推进，多个 Agent（雨娃、Talus、雅怡 Agent、77 Agent 等）需要在一个统一的协作空间内高效工作。今天的实践暴露了通信、触发、身份等基础设施问题，本规划旨在建立清晰的协作框架。

核心目标：

  1. 消除传话成本——Agent 之间能直接通信，不需要 Founder 当中介
  2. 明确决策权——每个产出由对应的人类决策者确认，Agent 不越权
  3. 版本可控——多 Agent 产出的文档有清晰的版本管理和汇总机制
  4. 可扩展——新 Agent 加入时能快速融入，不需要重新磨合

📡 通信协议

当前通信方式对比

方式 适用场景 优点 缺点
飞书群聊（人类 @） 激活 Agent、重大决策 能触发 Talus 响应 需要人类手动操作
飞书群聊（Agent 直接发） 日常交流、文档同步 无需人类参与 @ 为灰色，不会触发 Talus 通知
飞书私聊 一对一确认、敏感讨论 直接送达 其他 Agent 不可见
文档链接 复杂内容、需要 review 异步、可沉淀 实时性差
⚠️ 关键发现（今日实践）：
雨娃用 feishu_im_user_message 发群消息，显示身份为"雨娃-ECHO"应用，不是 zhouyating 个人。因此：
  • 雨娃发的 @Talus 是灰色，不会触发 Talus 响应
  • 只有 zhouyating 手动 @Talus，Talus 才会被激活参与
  • 激活后两个 Agent 可以在群里自主交流，但触发权在人类

建议的通信规范

场景 A：启动 Agent 间讨论

1. Founder 在群里发："@Talus @雨娃-ECHO 请讨论 XXX 问题"

2. Talus 被激活，雨娃看到后开始交流

3. 讨论中 Agent 自主交流，不需要人类参与

4. 产出文档后，@ 对应的人类决策者确认

场景 B：日常文档同步

1. Agent 直接发群消息（post 格式），附文档链接

2. 不需要 @ 任何人，其他 Agent 定期查看记录

3. 需要反馈时，再请求人类激活对方

场景 C：紧急沟通

1. 通过人类决策者私聊转达

2. 或直接电话联系（最高优先级）

🔄 协作流程

标准协作流程

需求提出 → 人类或 Agent 提出需求
分工决策 → Founder/Seaman 指定负责 Agent
执行 → Agent 独立或协作完成
产出 → 文档部署、链接同步
审阅 → 对应人类决策者确认
合并 → 雨娃汇总到主文档

决策权矩阵

决策内容 决策者 Agent 角色
协议架构方向 Seaman / 雅婷 Talus/雨娃 提议，Seaman 最终决策
产品功能优先级 雅怡 / 雅婷 Agent 执行，产品确认
模拟器技术实现 Seaman / Jason Agent 调研，技术决策
文档内容准确性 对应领域负责人 Agent 起草，专家审阅
部署与版本管理 雨娃（执行层面） 雨娃自主执行，异常情况上报

📁 文档管理

文档分级

级别 示例 管理责任 审阅要求
P0 - 核心协议 元规则协议 v3、权属蓝图、多Agent协作规划 Seaman 最终确认 必须人工审阅
P1 - 技术设计 ShiGraph 技术设计、模拟器架构 Seaman/Jason 确认 Agent 提议，人工确认
P2 - 调研产出 Talus 反馈摘要、Agent 模拟报告 雨娃汇总 异步 review
P3 - 过程记录 会议纪要、todo 列表 雨娃自主管理 定期同步即可

版本命名规范

[文档名]-v[主版本].[次版本]-[状态]

  • 主版本：重大结构变更（如 v2→v3）
  • 次版本：内容更新（如 v3.0→v3.1）
  • 状态：draft / review / final / deprecated

示例：echo-meta-rules-v3.1-draft.html

部署地址规范

  • 主版本：https://[branch].echo-v2.pages.dev/[path]
  • review 版本：https://[branch]-review.echo-v2.pages.dev/[path]
  • 临时版本：https://[branch]-temp.echo-v2.pages.dev/[path]
